# SmartSchool.bd — Documentation Index

This folder collects everything you need to navigate the SmartSchool.bd codebase. It is split into two layers:

- **Strategic docs** — phase plan, roadmap, history, decisions. Authored / maintained by the product owner.
- **Code-scan docs** — a faithful description of what is actually checked in at `HEAD`. Generated by reading the source tree directly.

When the two layers disagree, the **code-scan docs reflect what the code currently does** and the strategic docs reflect what we *want* it to do. Open a PR to reconcile.

## Top-level layout

```
/
├── .htaccess                # Apache/LiteSpeed front-controller rewrite into index.php
├── index.php                # CodeIgniter front controller
├── application/             # School app — controllers, models, views, libs, configs
├── system/                  # Stock CodeIgniter 3.1.13 framework (do not edit)
├── assets/                  # CSS / JS / images for both admin and public theme
├── uploads/                 # Runtime user uploads (photos, attachments, DB backups, etc.)
└── docs/                    # ← you are here
```

Inside `application/` the relevant directories are:

| Dir | Purpose |
|---|---|
| `controllers/` | 77 controllers — one per feature (Student, Fees, Saas, Signup, …) |
| `models/` | 71 models — DB access layer; matches controllers 1:1 in most cases |
| `views/` | View templates, organised by feature folder (`student/`, `saas/`, `home/`, …) |
| `core/` | `MY_Controller.php` (with `Admin_Controller`, `User_Controller`, `Authentication_Controller`, `Frontend_Controller` subclasses) + `MY_Model.php` |
| `libraries/` | Custom service classes (`App_lib`, payment gateways, SMS providers, PDF, QR, BulkBlue button, Zoom, etc.) |
| `helpers/` | `general_helper.php` (auth, translation, perms), `custom_fields_helper.php`, `saas_notify_helper.php`, `unzip_helper.php`, `general_helper_patch.php` |
| `config/` | `config.php`, `database.php`, `routes.php`, `autoload.php`, `migration.php`, `hooks.php`, `purchase_key.php`, `constants.php`, MIME / smiley / user-agent / doctype dictionaries |
| `language/english/` | i18n strings (other languages live in the DB `languages` table; see `general_helper::translate()`) |
| `third_party/` | Vendored Composer libs — mpdf, omnipay (PayPal, Stripe, Mollie, etc.), midtrans, razorpay, twilio, phpmailer, bigbluebutton |
| `migrations/` | **Empty** — only `index.html`. Schema lives in raw SQL dumps (see strategic docs) and ad-hoc one-shot migrations. |
| `hooks/` | **Empty** — only `index.html`. No CI hooks registered. |
| `cache/`, `logs/` | Runtime CI writable dirs. |

⚠️ **Repository hygiene observations** (called out in [`SECURITY.md`](SECURITY.md)):
- `application/config/database.php` contains plaintext DB credentials.
- `application/config/safe3.zip` is a backup snapshot of `config.php` + `purchase_key.php`.
- `application/bupviews.zip` is a snapshot of the `application/views/` tree.
- `application/views/landing/bak.zip` is a snapshot of older landing-page variants.
- `application/controllers/bak.*.php` and `application/models/bak.Saas_model.php` are pre-refactor backups still tracked in git.
- `uploads/db_backup/DB-backup_2025-01-11_23-10.zip` is a committed database backup.

These should be removed from version control and rotated where credentials are involved.

## How to keep docs current

When you change behaviour:
1. If you change strategy / roadmap → update `PROJECT_DRAFT.md` and/or `RESUME.md`.
2. If you add or rename a controller / model / library / route → update the matching code-scan doc (`CODE_MAP.md`, `ROUTES.md`, etc.).
3. If you flip a config flag, add an addon, or change autoload → update `CONFIGURATION.md`.

Code-scan docs were last regenerated on a full scan of `application/`, `system/`, `assets/`, `uploads/`, `index.php`, `.htaccess`. See each file's bottom-of-page note for the exact source paths it references.
