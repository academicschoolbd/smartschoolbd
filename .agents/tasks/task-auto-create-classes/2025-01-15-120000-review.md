# Auto-create Class 1-12 for all branches

A new `auto_create_classes()` endpoint in the Classes controller bulk-inserts Class 1 through Class 12 for every branch in the system. It checks for existing classes by `name_numeric + branch_id` to avoid duplicates, and allocates the first available section for each new class. The action is gated to super-admin only and triggered via a button in the classes index view with a JS confirmation prompt.

**Watch for:** The action mutates state via GET request, which CSRF protection won't cover and which browsers/crawlers can trigger unintentionally (confirmed). The bulk operation runs without a database transaction, so a mid-flight failure leaves partial data with no rollback (confirmed). The success message is not wrapped in `translate()`, breaking the i18n pattern used everywhere else in this codebase (confirmed).

## High-level view

The controller method iterates all branches, loops 1-12 for each, and inserts classes that don't already exist. Duplicate detection keys on `name_numeric` rather than `name`, so a manually-created class named "Class 5" with a different `name_numeric` would not be caught. Section allocation grabs the first section row for the branch without deterministic ordering.

The primary architectural concern is using a GET request for a write operation. CodeIgniter's CSRF protection only applies to POST requests, so this endpoint has no CSRF defence — any external page can trigger the action for a logged-in super-admin.

<summary>Details</summary>

## GET request for a write operation

The `auto_create_classes` action inserts rows into `class` and `sections_allocation` but is reachable via a plain GET link. CodeIgniter's CSRF middleware (enabled in this project: `$config['csrf_protection'] = TRUE`) only validates tokens on POST submissions. This means the endpoint has zero CSRF protection — an attacker can forge the request from any origin by embedding the URL in an image tag or redirect.

The idiomatic fix in this codebase is to use a `form_open()` POST form (which automatically includes the CSRF token) and check `$_POST` in the controller, matching the pattern already used by `index()` and `edit()`.

## No transaction boundary

The loop performs up to `branches * 12` insert operations. If the database connection drops or a constraint violation occurs partway through, some branches will have their classes created and others won't. The bulk nature of this operation amplifies the risk compared to single-class creation. `$this->db->trans_start()` before the outer loop and `$this->db->trans_complete()` after it would make the operation atomic.

## Non-deterministic section allocation

```php
$section = $this->db->where('branch_id', $branch->id)->limit(1)->get('section')->row();
```

Without an `ORDER BY`, the "first" section depends on the storage engine's physical row order, which can change after deletes or optimizations. Adding `->order_by('id', 'asc')` makes the choice deterministic and predictable across runs.

## Hardcoded English string

Every other call to `set_alert` in this controller passes the message through `translate()`. This one concatenates a raw English string, which will display English text regardless of the user's locale setting:

```php
set_alert('success', 'Total ' . $count . ' classes have been created successfully across all branches.');
```

## Duplicate detection scope

The check uses `name_numeric` as the deduplication key:

```php
$exists = $this->db->get_where('class', array('name_numeric' => $i, 'branch_id' => $branch->id))->num_rows();
```

If a branch has a manually-created class where `name_numeric` was left empty or set to a different value but the display name is "Class 5", the auto-create will insert a second "Class 5" with `name_numeric = 5`. The UI would then show two rows with the same display name for that branch.
